O que é um Banco de Dados?
Entendendo o conceito antes de tudo
Analogia: O Armário Organizado

Imagine um armário gigante de escritório com várias gavetas. Cada gaveta tem uma etiqueta (nome da tabela): "Fornecedores", "Produtos", "Clientes". Dentro de cada gaveta tem fichas organizadas em linhas e colunas. Isso é um banco de dados relacional! 📂

📖 Definição

Um banco de dados é um lugar organizado para armazenar, consultar e manipular dados. Quando você fecha o programa, os dados não somem — ficam salvos permanentemente. É diferente de variáveis no código, que morrem quando o programa fecha.

🗄️Banco (Database)O armário inteiro
📋Tabela (Table)Uma gaveta — ex: "fornecedor"
📊Coluna (Column)Os campos — ex: "nome", "email"
📝Linha (Row)Um registro — ex: "Distribuidora ABC"
🔑Chave Primária (PK)ID único de cada registro
📝SQLA "linguagem" para falar com o banco

Instalando o PostgreSQL com Docker
A forma mais fácil e profissional de rodar bancos de dados
🐳 O que é Docker?

Docker é como uma caixa mágica que empacota um programa com tudo que ele precisa para funcionar. Em vez de instalar o PostgreSQL direto no seu computador (e lidar com configurações complicadas), você cria um container — uma "mini-máquina" que roda o PostgreSQL isoladamente.

Analogia: Container = Marmita

Instalar um programa direto no computador é como cozinhar na sua cozinha — suja tudo, ocupa espaço, e é difícil de limpar depois. Usar Docker é como pedir uma marmita: vem tudo pronto, embalado, e quando você termina, joga fora a embalagem sem sujar nada! 🍱

📥 Passo 1 — Instale o Docker Desktop

O Docker Desktop é o app que permite usar Docker no seu computador de forma visual e simples.

  1. Acesse docker.com/products/docker-desktop
  2. Baixe para o seu sistema operacional (Windows, Mac ou Linux)
  3. Instale normalmente (next, next, finish...)
  4. Abra o Docker Desktop e aguarde ele iniciar (ícone da baleia 🐳 fica verde)
✅
Como saber se o Docker está funcionando? Abra o terminal e digite docker --version. Se aparecer uma versão, está tudo certo!

🚀 Passo 2 — Criar o Container do PostgreSQL

Com o Docker Desktop rodando, abra o terminal (Prompt de Comando, PowerShell ou Terminal do Mac) e cole este comando:

Terminalbash
docker run --name postgres-fullstack -e POSTGRES_PASSWORD=123 -e POSTGRES_USER=postgres -p 5432:5432 -d postgres
🔍 O que cada parte desse comando faz? ▼
docker run — Cria e inicia um novo container
--name postgres-fullstack — Dá o nome "postgres-fullstack" ao container (pode ser qualquer nome)
-e POSTGRES_PASSWORD=123 — Define a senha do usuário postgres como 123
-e POSTGRES_USER=postgres — Define o usuário como postgres
-p 5432:5432 — Mapeia a porta 5432 do container para a porta 5432 do seu PC (porta padrão do PostgreSQL)
-d — Roda em background (não trava o terminal)
postgres — Nome da imagem oficial do PostgreSQL no Docker Hub
🐳
Pronto! O PostgreSQL já está rodando! Você pode ver o container no Docker Desktop (aba "Containers") — ele deve estar com status verde Running.

🔧 Comandos úteis do Docker

docker ps
Lista containers rodando
docker stop postgres-fullstack
Para o container
docker start postgres-fullstack
Inicia novamente
docker rm postgres-fullstack
Remove o container (dados perdem!)
⚠️
Atenção: Sempre que ligar o computador, abra o Docker Desktop primeiro! Sem ele rodando, o PostgreSQL não funciona. O container não inicia sozinho — você pode configurar auto-start no Docker Desktop ou usar docker start postgres-fullstack.

DBeaver — Sua IDE para Banco de Dados
Visualize e manipule o banco de forma gráfica
🤔 O que é o DBeaver?

O DBeaver é como o VS Code, mas para bancos de dados. É uma ferramenta gráfica (GUI) que permite ver tabelas, escrever SQL, e manipular dados de forma visual — sem precisar decorar comandos. É gratuito e funciona com qualquer banco!

Analogia: Explorer de Arquivos

O DBeaver é para o banco de dados o que o Explorer/Finder é para arquivos no computador. Você pode navegar pelas "pastas" (bancos), abrir "arquivos" (tabelas), e editar o conteúdo visualmente. Sem ele, seria como usar o computador só pelo terminal — funciona, mas é mais difícil! 🖥️

📥 Instalando o DBeaver

  1. Acesse dbeaver.io/download
  2. Baixe o DBeaver Community (versão gratuita)
  3. Instale normalmente

SQL Básico — Falando com o Banco

🗣️ O que é SQL?

SQL (Structured Query Language) é a linguagem que usamos para conversar com o banco de dados. É como uma "língua" que o banco entende. Você escreve comandos em SQL e o banco executa — cria tabelas, insere dados, busca informações, etc.

Analogia: Cardápio do Restaurante

SQL é como fazer pedidos em um restaurante. Você tem CREATE TABLE = "quero criar um novo prato no cardápio", INSERT = "quero servir um prato", SELECT = "me mostra o que tem", UPDATE = "muda o prato", DELETE = "tire esse prato". Simples assim! 🍽️

📌
Onde escrever SQL? No DBeaver! Conecte-se ao banco bd_sistema_gestao, clique no botão "SQL" (ou Ctrl+Enter / Cmd+Enter) para abrir o editor SQL. Escreva os comandos lá e execute com Ctrl+Enter (ou o botão ▶️).

CREATE TABLE teste_fornecedor (

SELECT Consultando dados

O comando mais usado do SQL! É como perguntar ao banco: "me mostra os dados":

DBeaver — SQL EditorSQL
-- Buscar TODOS os fornecedores (todas as colunas)
SELECT * FROM teste_fornecedor;

-- Buscar só nome e cidade
SELECT nome, cidade FROM teste_fornecedor;

-- Buscar fornecedor com id = 1
SELECT * FROM teste_fornecedor WHERE id = 1;

-- Buscar fornecedores de São Paulo
SELECT * FROM teste_fornecedor WHERE cidade = 'São Paulo';
🔍 Traduzindo cada SELECT: ▼
SELECT * FROM teste_fornecedor;
→ "Me dá tudo (*) que tem na tabela teste_fornecedor"
SELECT nome, cidade FROM teste_fornecedor;
→ "Me dá só o nome e a cidade dos fornecedores"
SELECT * FROM teste_fornecedor WHERE id = 1;
→ "Me dá tudo do fornecedor cujo id é 1"
SELECT * FROM teste_fornecedor WHERE cidade = 'São Paulo';
→ "Me dá todos que são de São Paulo"

UPDATE Atualizando dados

Precisa corrigir ou atualizar algum dado? Use o UPDATE:

DBeaver — SQL EditorSQL
-- Alterar o email do fornecedor com id = 1
UPDATE teste_fornecedor
SET email = 'novoemail@abc.com'
WHERE id = 1;

-- Verificar a alteração
SELECT * FROM teste_fornecedor WHERE id = 1;
🚨
NUNCA esqueça o WHERE no UPDATE! Se fizer UPDATE teste_fornecedor SET email = 'teste@teste.com' sem WHERE, TODOS os fornecedores terão o email alterado! O WHERE filtra quem será afetado.

DELETE Deletando dados

Para remover um registro da tabela:

DBeaver — SQL EditorSQL
-- Deletar o fornecedor com id = 3
DELETE FROM teste_fornecedor
WHERE id = 3;

-- Verificar: agora só devem aparecer 2 registros
SELECT * FROM teste_fornecedor;
🚨
NUNCA esqueça o WHERE no DELETE! Um DELETE FROM teste_fornecedor sem WHERE apaga TODOS os dados da tabela. Cuidado!

📢 Avisos importantes para a prática

⚠️
Sempre use WHERE em UPDATE e DELETE quando a intenção for alterar ou remover apenas um registro. Sem o WHERE, o comando afeta todas as linhas da tabela — e não tem "Ctrl+Z" no banco de dados!
🔍
Confira em qual database você está conectado! No DBeaver, olhe o painel esquerdo — o banco ativo aparece em negrito. Se estiver no banco postgres em vez do bd_sistema_gestao, seus comandos não vão funcionar como esperado.
🏋️
Pratique executando, não apenas lendo! SQL se aprende fazendo. Cada comando aqui deve ser digitado e executado no DBeaver. Leitura passiva não fixa — é a prática que constrói a memória.

Erros Mais Comuns
Problemas que todo iniciante encontra (e como resolver!)

Não se preocupe se algo der errado — isso é parte normal do aprendizado! Aqui estão os erros mais comuns e como resolver cada um:

🐳
Docker Desktop não está aberto

Sintoma: Ao rodar docker run... aparece erro "Cannot connect to the Docker daemon".

Solução: Abra o Docker Desktop e espere o ícone da baleia 🐳 ficar verde. Depois tente o comando de novo.

🔌
Porta 5432 já em uso

Sintoma: Erro "port is already allocated" ou "Bind for 0.0.0.0:5432 failed".

Solução: Já existe algo usando a porta 5432. Pode ser outro container ou um PostgreSQL instalado localmente. Pare o outro serviço ou mude a porta: -p 5433:5432 (e use 5433 no DBeaver).

🔑
Senha ou usuário incorretos no DBeaver

Sintoma: "FATAL: password authentication failed" ao testar a conexão.

Solução: Confira se o usuário é postgres e a senha é 123 (exatamente como definimos no docker run). Letras maiúsculas/minúsculas importam!

🔄
Esquecer de dar Refresh no DBeaver

Sintoma: Você criou o banco ou a tabela mas não aparece no painel esquerdo.

Solução: Clique com botão direito na conexão e selecione "Refresh". O DBeaver não atualiza sozinho!

🎯
Estar conectado no banco errado

Sintoma: "ERROR: relation "teste_fornecedor" does not exist" — mas você acabou de criar!

Solução: Verifique se está no database bd_sistema_gestao e não no postgres (padrão). Dê duplo-clique no banco correto no painel esquerdo.

💀
UPDATE ou DELETE sem WHERE — PERIGO!

Sintoma: Todos os registros da tabela foram alterados ou apagados.

Solução: Infelizmente, não tem "desfazer" fácil. Sempre revise o comando antes de executar. Se aconteceu, apague a tabela com DROP TABLE e recrie tudo.

🧠
Dica de ouro: Quando algo dá errado, leia a mensagem de erro com calma. Ela quase sempre diz o que aconteceu. Copie o texto do erro e pesquise no Google — você vai se surpreender como é rápido encontrar a solução!

import org.springframework.stereotype.Service;

import java.util.Optional;

@Service  // "Esta classe é um Service do Spring!"@ServiceMarca a classe como um componente de serviço. O Spring cria uma instância automaticamente.
public class FornecedorService {

    @Autowired  // 💉 Injeção de dependência!@AutowiredO Spring injeta automaticamente uma instância do FornecedorRepository aqui. Você NÃO precisa fazer new FornecedorRepository()!
    FornecedorRepository fornecedorRepository;

    // CREATE — salvar novo fornecedor
    public void criarFornecedor(Fornecedor fornecedor) {
        fornecedorRepository.save(fornecedor);
    }

    // READ — listar todos os fornecedores
    public List<Fornecedor> listarTodosFornecedores() {
        return fornecedorRepository.findAll();
    }

    // READ — buscar por ID
    public Optional<Fornecedor> buscarFornecedorPorId(Long id) {
        return fornecedorRepository.findById(id);
    }

    // UPDATE — atualizar fornecedor por ID
    public void atualizarFornecedorPorId(Long id, Fornecedor fornecedor) {
        Optional<Fornecedor> fornecedorDoBanco = buscarFornecedorPorId(id);

        if (fornecedorDoBanco.isEmpty()) {
            throw new ResponseStatusException(
                HttpStatus.NOT_FOUND, "Fornecedor não encontrado"
            );
        }

        Fornecedor fornecedorParaEditar = fornecedorDoBanco.get();
        fornecedorParaEditar.setNome(fornecedor.getNome());
        fornecedorParaEditar.setCnpj(fornecedor.getCnpj());
        fornecedorParaEditar.setEmail(fornecedor.getEmail());
        fornecedorParaEditar.setTelefone(fornecedor.getTelefone());
        fornecedorParaEditar.setCep(fornecedor.getCep());
        fornecedorParaEditar.setLogradouro(fornecedor.getLogradouro());
        fornecedorParaEditar.setNumero(fornecedor.getNumero());
        fornecedorParaEditar.setBairro(fornecedor.getBairro());
        fornecedorParaEditar.setCidade(fornecedor.getCidade());
        fornecedorParaEditar.setUf(fornecedor.getUf());

        fornecedorRepository.save(fornecedorParaEditar);
    }

    // DELETE — deletar fornecedor por ID

🔍 O que é Optional e por que usamos? ▼

Optional<Fornecedor> é como uma caixa de presente que pode estar vazia ou ter um fornecedor dentro.

Quando buscamos por ID, o fornecedor pode não existir. O Optional nos obriga a tratar esse caso:

  • .isEmpty() → a caixa está vazia? (fornecedor não existe)
  • .get() → pega o fornecedor de dentro da caixa
  • .isPresent() → tem algo na caixa?

Isso evita o famoso NullPointerException! 💀

📦 DTO — Data Transfer Object

O conceito mais importante desta aula!

🤔 O que é um DTO?

DTO é uma classe simples que carrega dados de um lugar para outro. Ela serve como um "filtro" entre o que o front envia e o que o banco armazena. O front nunca conversa diretamente com a entidade do banco!

Analogia: O Formulário vs A Ficha Interna

Imagine um hotel. Quando você chega, preenche um formulário na recepção com nome, documento e quantos dias vai ficar. Esse formulário é o RequestDTO — só tem as informações que o hóspede precisa fornecer.

Internamente, o hotel tem uma ficha completa com número do quarto, data de check-in, valor total, status do pagamento. Essa ficha interna é a Entidade (Model).

Quando o hotel mostra o resumo da reserva, ele não mostra TUDO da ficha interna — mostra só o necessário em um comprovante bonito. Esse comprovante é o ResponseDTO! 🏨

❌ Sem DTO (como fizemos no Fornecedor):

O front envia a entidade inteira (Fornecedor.java). Problemas:

❌ O front pode mandar o campo id e tentar sobrescrever
❌ Se a entidade mudar (nova coluna), a API quebra
❌ Expõe a estrutura interna do banco para quem consome
❌ Não tem validação automática dos campos

✅ Com DTO (como vamos fazer em Produto):

O front envia um DTO enxuto, com validações. Vantagens:

✅ O front só envia o que precisa (sem id, sem dados internos)
✅ Validações automáticas (@NotBlank, @Positive) antes de chegar no Service
✅ A entidade pode mudar sem quebrar a API
✅ O response mostra só o que o front precisa ver

📝 Java Record — DTOs Modernos

🤔 O que é um Record?

O record é uma forma moderna e enxuta do Java para criar classes que só carregam dados. É perfeito para DTOs! Em vez de escrever uma classe com atributos + getters + construtor + toString (umas 30 linhas), você escreve 3 linhas.

❌ Classe tradicional (muitas linhas):

Java 
// Classe tradicional para DTO — MUITO código repetitivo!
public class ProdutoRequestDTO {
    private String nome;
    private BigDecimal preco;

    // Construtor
    public ProdutoRequestDTO(String nome, BigDecimal preco) {
        this.nome = nome;
        this.preco = preco;
    }

    // Getters
    public String getNome() { return nome; }
    public BigDecimal getPreco() { return preco; }

    // toString, equals, hashCode... mais 20 linhas 😴
}

✅ Com Record (3 linhas!):

Java 
// Record — faz TUDO que a classe acima faz, em 3 linhas!
public record ProdutoRequestDTO(
    String nome,
    BigDecimal preco
) {}
💡
Diferença importante: No record, para acessar os dados, usamos dto.nome() ao invés de dto.getNome(). Sem o "get" na frente! O record gera isso automaticamente.

🔄 Mapper — O Tradutor

🤔 O que é um Mapper?

O Mapper é uma classe que converte objetos de um tipo para outro. No nosso caso, converte RequestDTO → Entidade (quando recebe dados do front) e Entidade → ResponseDTO (quando devolve dados pro front).

Analogia: O Tradutor de Idiomas

Imagine dois países que falam idiomas diferentes: o País do Front (fala "DTO") e o País do Banco (fala "Entidade"). O Mapper é o tradutor que fica na fronteira convertendo de uma língua para outra. Quando o front manda uma mensagem (DTO), o tradutor converte para a língua do banco (Entidade). Quando o banco responde, o tradutor converte de volta para o front! 🌐

📋 ServiceImpl — Interface + Implementação

🤔 O que é essa separação?

No Fornecedor, tínhamos apenas FornecedorService (uma classe só). Agora vamos separar em duas coisas: uma interface (o contrato, que diz "quais métodos existem") e uma implementação (a classe que realmente faz o trabalho).

✅ Bean Validation — Validações Automáticas

🤔 O que é Bean Validation?

Bean Validation permite validar os dados automaticamente usando anotações. Em vez de escrever if (nome == null) throw ... na mão, você simplesmente coloca @NotBlank no campo e o Spring valida sozinho!

Analogia: O Segurança da Porta

Sem validação, é como uma festa sem segurança na porta — qualquer pessoa entra, com dados faltando, com preço negativo, com email inválido. Com Bean Validation, você coloca um segurança que confere cada campo antes de deixar entrar! Se algo estiver errado, o segurança barra e devolve uma mensagem dizendo o que falta. 🚪👮

🏷️ Anotações que vamos usar:

@NotBlank
Texto não pode ser vazio nem só espaços. Ideal para nomes e descrições.
@NotNull
O campo é obrigatório — não pode ser null. Para números, enums e objetos.
@Positive
O número deve ser maior que zero. Perfeito para preço!
@PositiveOrZero
Zero ou positivo. Ideal para estoque (pode ter zero em estoque).
⚡
Detalhe importante: As validações só funcionam se colocar @Valid no parâmetro do Controller! Sem o @Valid, as anotações ficam lá mas não fazem nada. É como ligar o segurança mas esquecer de pedir pra ele conferir os ingressos!

💉 @RequiredArgsConstructor — Adeus @Autowired

🤔 O que muda na injeção de dependência?

No Fornecedor, usamos @Autowired para injetar o Repository no Service. Funciona, mas o Spring recomenda injeção pelo construtor como melhor prática. O @RequiredArgsConstructor do Lombok gera esse construtor automaticamente!

❌ Antes (Fornecedor)
@Autowired
FornecedorRepository repo;
✅ Agora (Produto)
@RequiredArgsConstructor
private final ProdutoRepository repo;
💡
O segredo é o private final! O @RequiredArgsConstructor gera um construtor com todos os atributos que são final. O Spring vê esse construtor e injeta automaticamente. Resultado: mesma funcionalidade, código mais limpo!

⚡ Lambda e Stream — Simplificando Código

🤔 O que é Lambda?

Lambda é uma forma de escrever uma função pequena em uma linha. Em vez de criar um método inteiro, você escreve (parametro) -> ação. Vamos usar em dois lugares: no orElseThrow e no stream().map().

1️⃣ Lambda no orElseThrow

Analogia: O Entregador

Quando você pede uma pizza, o entregador vai no endereço. Se encontra alguém, entrega a pizza (retorna o produto). Se não encontra ninguém, ele liga reclamando: "não tem ninguém aqui!" (lança o erro). O orElseThrow funciona assim: "busca no banco, se não achar, reclama!" 🍕

❌ Antes (6 linhas)
Optional<Produto> opt = repo.findById(id);

if (opt.isEmpty()) {
  throw new ResponseStatusException(...);
}

Produto produto = opt.get();
✅ Agora (1 linha!)
Produto produto = repo.findById(id)
  .orElseThrow(() -> new
    ResponseStatusException(...));

O () -> é a lambda!
"Se não achar, execute isso"

🔬 Decompondo a lambda:

repo.findById(id) → busca no banco pelo ID
.orElseThrow( → se não encontrou...
  () -> → a lambda! "execute o que vem depois da seta"
  new ResponseStatusException(NOT_FOUND, "...") → lança erro 404
);

2️⃣ Stream + Map — Convertendo Listas

Analogia: A Esteira de Fábrica

Imagine uma esteira de fábrica 🏭. Você coloca laranjas (entidades) na esteira, e no meio do caminho tem uma máquina de suco (o Mapper) que transforma cada laranja em suco (DTO). No final da esteira, sai uma lista de sucos! Isso é o stream().map().toList()!

❌ Sem Stream (5 linhas)
List<Produto> produtos = repo.findAll();
List<ResponseDTO> lista = new ArrayList<>();

for (Produto p : produtos) {
  lista.add(Mapper.toResponseDTO(p));
}

return lista;
✅ Com Stream (1 return!)
return repo.findAll()
  .stream()
  .map(ProdutoMapper::toResponseDTO)
  .toList();

🔬 Decompondo o Stream:

repo.findAll() → busca todos do banco (lista de Produtos)
.stream() → liga a esteira! Transforma a lista num fluxo processável
.map(ProdutoMapper::toResponseDTO) → a máquina! Aplica a conversão em cada item
.toList() → empacota! Junta tudo numa lista de DTOs
💡
O ProdutoMapper::toResponseDTO é chamado de method reference. É a mesma coisa que produto -> ProdutoMapper.toResponseDTO(produto), só que mais curto. O Java e o IntelliJ preferem essa forma!

🔗 @ManyToOne — Relacionamento entre Tabelas

🤔 O que é @ManyToOne?

Essa anotação diz que MUITOS produtos pertencem a UM fornecedor. No banco, isso cria uma chave estrangeira (foreign key) — uma coluna fornecedor_id na tabela produto que aponta para a tabela fornecedor.

Analogia: Funcionários e Empresa

Uma empresa (fornecedor) tem muitos funcionários (produtos). Cada funcionário pertence a uma empresa só. Na ficha de cada funcionário, tem anotado "empresa: XYZ" — esse é o fornecedor_id! 🏢👥

🗄️ Como fica no banco:

📋 fornecedor

id 🔑nomefornecedor_id 🔗

☝️ Os produtos 1 e 2 pertencem ao fornecedor 1 (Dist. ABC). O produto 3 pertence ao fornecedor 2 (Forn. XYZ). A coluna fornecedor_id é a chave estrangeira que cria esse vínculo!

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

import java.math.BigDecimal;

@NoArgsConstructor
@AllArgsConstructor
@Data
@Table(name = "produto")
@Entity
public class Produto {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "nome", nullable = false)
    private String nome;

    @Column(name = "descricao")
    private String descricao;

    @Column(name = "preco", nullable = false)
    private BigDecimal preco;

    @Column(name = "quantidade_estoque", nullable = false)
    private Integer quantidadeEstoque;

    @Enumerated(EnumType.STRING)  // Salva o NOME do enum (ex: "INFORMATICA")@Enumerated(EnumType.STRING)Salva o nome do enum como texto no banco. Sem isso, salvaria o número da posição (0, 1, 2...) — que é confuso e frágil.
    @Column(name = "categoria", nullable = false)
    private CategoriaProdutoEnum categoria;

    @ManyToOne  // MUITOS produtos para UM fornecedor@ManyToOneDefine o relacionamento: muitos Produtos pertencem a um Fornecedor. Cria a coluna fornecedor_id como chave estrangeira.

}

public class ProdutoMapper {

    // Converte RequestDTO → Entidade
    // Precisa do Fornecedor porque o DTO só tem o ID
    public static Produto toEntity(ProdutoRequestDTO dto, Fornecedor fornecedor) {

        Produto produto = new Produto();

        produto.setNome(dto.nome());          // record usa .nome() sem "get"!
        produto.setDescricao(dto.descricao());
        produto.setPreco(dto.preco());
        produto.setQuantidadeEstoque(dto.quantidadeEstoque());
        produto.setCategoria(dto.categoria());
        produto.setFornecedor(fornecedor);    // objeto completo, não o ID!

        return produto;
    }

    // Converte Entidade → ResponseDTO
    public static ProdutoResponseDTO toResponseDTO(Produto produto) {

        ProdutoResponseDTO dto = new ProdutoResponseDTO(
                produto.getId(),
                produto.getNome(),
                produto.getDescricao(),
                produto.getPreco(),
                produto.getQuantidadeEstoque(),
                produto.getCategoria(),
                produto.getFornecedor().getId(),          // pega o ID do fornecedor
                produto.getFornecedor().getNomeFantasia() // pega o nome do fornecedor
        );

        return dto;
    }
}

package br.com.sistemagestao.api.service;

import br.com.sistemagestao.api.dto.produto.ProdutoRequestDTO;
import br.com.sistemagestao.api.dto.produto.ProdutoResponseDTO;
import br.com.sistemagestao.api.mapper.ProdutoMapper;
import br.com.sistemagestao.api.model.Fornecedor;
import br.com.sistemagestao.api.model.Produto;
import br.com.sistemagestao.api.repository.FornecedorRepository;
import br.com.sistemagestao.api.repository.ProdutoRepository;
import br.com.sistemagestao.api.service.ProdutoService;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;

import java.util.List;

@Service
@RequiredArgsConstructor  // Lombok gera o construtor com os campos "final"@RequiredArgsConstructorO Lombok gera um construtor com TODOS os campos marcados como final. O Spring usa esse construtor para injetar automaticamente. Substitui o @Autowired!
public class ProdutoServiceImpl implements ProdutoService {

    // Injeção de dependência pelo construtor (via Lombok)
    private final ProdutoRepository produtoRepository;
    private final FornecedorRepository fornecedorRepository;

    @Override
    public ProdutoResponseDTO criarProduto(ProdutoRequestDTO dto) {

        // Busca o fornecedor pelo ID. Se não existir, lança 404.
        Fornecedor fornecedor = fornecedorRepository.findById(dto.fornecedorId())
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Fornecedor não encontrado"));

        // Converte DTO → Entidade usando o Mapper
        Produto produto = ProdutoMapper.toEntity(dto, fornecedor);
        produtoRepository.save(produto);

        // Converte Entidade → ResponseDTO e retorna
        return ProdutoMapper.toResponseDTO(produto);
    }

    @Override
    public List<ProdutoResponseDTO> listarTodosProdutos() {

        // Busca todos → stream → converte cada um → lista de DTOs
        return produtoRepository.findAll()
                .stream()
                .map(ProdutoMapper::toResponseDTO)
                .toList();
    }

    @Override
    public ProdutoResponseDTO buscarProdutoPorId(Long id) {

        Produto produto = produtoRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Produto não encontrado"));

        return ProdutoMapper.toResponseDTO(produto);
    }

    @Override
    public ProdutoResponseDTO atualizarProdutoPorId(Long id, ProdutoRequestDTO dto) {

        Produto produto = produtoRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Produto não encontrado"));

        Fornecedor fornecedor = fornecedorRepository.findById(dto.fornecedorId())
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Fornecedor não encontrado"));

        produto.setNome(dto.nome());
        produto.setDescricao(dto.descricao());
        produto.setPreco(dto.preco());
        produto.setQuantidadeEstoque(dto.quantidadeEstoque());
        produto.setCategoria(dto.categoria());
        produto.setFornecedor(fornecedor);

        produtoRepository.save(produto);

        return ProdutoMapper.toResponseDTO(produto);
    }

    @Override
    public void deletarProdutoPorId(Long id) {

        Produto produto = produtoRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Produto não encontrado"));

        produtoRepository.deleteById(produto.getId());
    }
}

package br.com.petshop.api.service;

import br.com.petshop.api.dto.tutor.TutorRequestDTO;
import br.com.petshop.api.dto.tutor.TutorResponseDTO;
import br.com.petshop.api.mapper.TutorMapper;
import br.com.petshop.api.model.Tutor;
import br.com.petshop.api.repository.TutorRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;

import java.util.List;

@Service
@RequiredArgsConstructor
public class TutorServiceImpl implements TutorService {

    private final TutorRepository tutorRepository;

    @Override
    public TutorResponseDTO criarTutor(TutorRequestDTO dto) {

        Tutor tutor = TutorMapper.toEntity(dto);
        tutorRepository.save(tutor);

        return TutorMapper.toResponseDTO(tutor);
    }

    @Override
    public List<TutorResponseDTO> listarTodosTutores() {

        return tutorRepository.findAll()
                .stream()
                .map(TutorMapper::toResponseDTO)
                .toList();
    }

    @Override
    public TutorResponseDTO buscarTutorPorId(Long id) {

        Tutor tutor = tutorRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Tutor não encontrado"));

        return TutorMapper.toResponseDTO(tutor);
    }

    @Override
    public TutorResponseDTO atualizarTutorPorId(Long id, TutorRequestDTO dto) {

        Tutor tutor = tutorRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Tutor não encontrado"));

        tutor.setNomeCompleto(dto.nomeCompleto());
        tutor.setEmail(dto.email());
        tutor.setTelefone(dto.telefone());

        tutorRepository.save(tutor);

        return TutorMapper.toResponseDTO(tutor);
    }

    @Override
    public void deletarTutorPorId(Long id) {

        Tutor tutor = tutorRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Tutor não encontrado"));

        tutorRepository.deleteById(tutor.getId());
    }
}

package br.com.petshop.api.service;

import br.com.petshop.api.dto.pet.PetRequestDTO;
import br.com.petshop.api.dto.pet.PetResponseDTO;
import br.com.petshop.api.mapper.PetMapper;
import br.com.petshop.api.model.Pet;
import br.com.petshop.api.model.Tutor;
import br.com.petshop.api.repository.PetRepository;
import br.com.petshop.api.repository.TutorRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;

import java.util.List;

@Service
@RequiredArgsConstructor
public class PetServiceImpl implements PetService {

    private final PetRepository petRepository;
    private final TutorRepository tutorRepository;

    @Override
    public PetResponseDTO criarPet(PetRequestDTO dto) {

        Tutor tutor = tutorRepository.findById(dto.tutorId())
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Tutor não encontrado"));

        Pet pet = PetMapper.toEntity(dto, tutor);
        petRepository.save(pet);

        return PetMapper.toResponseDTO(pet);
    }

    @Override
    public List<PetResponseDTO> listarTodosPets() {

        return petRepository.findAll()
                .stream()
                .map(PetMapper::toResponseDTO)
                .toList();
    }

    @Override
    public PetResponseDTO buscarPetPorId(Long id) {

        Pet pet = petRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Pet não encontrado"));

        return PetMapper.toResponseDTO(pet);
    }

    @Override
    public PetResponseDTO atualizarPetPorId(Long id, PetRequestDTO dto) {

        Pet pet = petRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Pet não encontrado"));

        Tutor tutor = tutorRepository.findById(dto.tutorId())
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Tutor não encontrado"));

        pet.setNome(dto.nome());
        pet.setRaca(dto.raca());
        pet.setIdade(dto.idade());
        pet.setEspecie(dto.especie());
        pet.setTutor(tutor);

        petRepository.save(pet);

        return PetMapper.toResponseDTO(pet);
    }

    @Override
    public void deletarPetPorId(Long id) {

        Pet pet = petRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Pet não encontrado"));

        petRepository.deleteById(pet.getId());
    }
}

🔗 @OneToOne — Endereço como entidade separada

O conceito mais importante deste capítulo!

🤔 O que é @OneToOne?

É a anotação que diz: UM cliente tem UM endereço, e UM endereço pertence a UM cliente. No banco, isso cria uma chave estrangeira (FK) na tabela cliente que aponta para a tabela endereco. As duas tabelas existem separadas — uma linkada à outra!

Analogia: A Pessoa e o seu CPF

Pense numa pessoa e seu CPF. Cada pessoa tem um único CPF, e cada CPF pertence a uma única pessoa. Eles existem como informações separadas (a Receita Federal guarda os CPFs em uma "tabela", a base de cidadãos em outra), mas estão linkados — um aponta pro outro. Esse é o relacionamento @OneToOne! 🪪

📦 Antes (Fornecedor — endereço flat):

No Capítulo 1, o Fornecedor tem os campos de endereço direto na entidade:

Fornecedor { id, nome, cnpj, email, cep, logradouro, numero, bairro, cidade, uf }

É simples e funciona, mas duplica a estrutura de endereço em todo lugar que precisar dela. 😕

✅ Agora (Cliente — endereço como entidade):

O Cliente terá uma referência a um Endereço em outra tabela:

Cliente { id, nomeCompleto, cpf, email, telefone, dataNascimento, endereco → Endereco }
Endereco { id, cep, logradouro, numero, complemento, bairro, cidade, uf }

Estrutura reutilizável! No próximo capítulo o Fornecedor vai usar essa mesma entidade. ✨

🌊 cascade + orphanRemoval — Ciclo de vida acoplado

🤔 Pra que servem?

Quando você tem dois objetos linkados (Cliente ↔ Endereço), surge a pergunta: "quando eu salvar o Cliente, o Endereço também é salvo?" e "quando eu deletar o Cliente, o Endereço também é apagado?". Por padrão, o JPA não faz nada disso — você teria que salvar e deletar cada um manualmente! O cascade e o orphanRemoval resolvem isso.

Analogia: O Caderno e a Capa

Imagine um caderno com sua capa personalizada. Quando você compra o caderno, a capa vem junto (não compra separado). Quando joga o caderno fora, a capa vai junto (não vai guardar uma capa solta na gaveta!). Essa é a ideia do cascade = ALL e orphanRemoval = true: a vida do Endereço está colada à vida do Cliente. 📓

🔬 Decompondo:

cascade = CascadeType.ALL

"Toda operação que eu fizer no Cliente — salvar, atualizar, deletar — propaga para o Endereço também."

orphanRemoval = true

"Se o Endereço ficar órfão (sem cliente apontando pra ele), apaga ele do banco automaticamente."

⚡
Resultado prático: Você só vai precisar de clienteRepository.save(cliente) e o JPA salva o Endereço junto. Você só vai precisar de clienteRepository.deleteById(id) e o Endereço some também. Não precisa nem de EnderecoRepository nesse capítulo! 🎉

📦 DTO Aninhado — DTO dentro de DTO

🤔 O que é um DTO aninhado?

É um DTO que contém outro DTO dentro dele. No nosso caso, o ClienteRequestDTO vai ter um campo do tipo EnderecoDTO. Isso reflete o JSON aninhado que o front vai enviar — onde o endereço é um sub-objeto.

Analogia: O Formulário com Sub-formulário

Já preencheu um cadastro online onde tem o formulário principal (nome, CPF, e-mail) e, no meio dele, um bloco separado de "Endereço" com seus próprios campos? Esse bloco é o sub-formulário! O EnderecoDTO dentro do ClienteRequestDTO é exatamente isso. 📝

✅ Validações Novas — @CPF, @Email, @Past

🤔 Mais anotações prontas?

O Bean Validation tem muitas anotações prontas além das que vimos no Produto (@NotBlank, @Positive...). Para Cliente, vamos usar três novas que cobrem casos super comuns no mundo real!

Analogia: Os Especialistas da Portaria

No Produto vimos o segurança que confere coisas básicas (não está vazio, é positivo). Agora chegaram três especialistas na portaria: o despachante que confere CPFs, o analista de e-mail que valida formato, e o cronista que confere se a data é mesmo do passado! 🕵️‍♂️

🏷️ Anotações novas:

@CPF
Valida se a string é um CPF brasileiro válido (algoritmo dos dígitos verificadores). Aceita com ou sem máscara!
@Email
Valida o formato de e-mail (tem @, tem domínio, etc). Não confere se o e-mail existe de verdade — só o formato.
@Past
A data deve ser anterior à data atual. Perfeito para data de nascimento (ninguém nasce no futuro!).
📦
Importante: O @CPF vem do pacote org.hibernate.validator.constraints.br.CPF — é uma extensão BR do Hibernate Validator. Mas já vem incluído com a dependência spring-boot-starter-validation que adicionamos no Produto! Não precisa instalar nada novo. 🎉

Compare com o @ManyToOne do Produto: Lá usamos @ManyToOne porque vários produtos apontam pro mesmo fornecedor. Aqui usamos @OneToOne porque cada cliente tem seu próprio endereço, não compartilhado!

public class ClienteMapper {

    // ───── CLIENTE ─────

    // Converte RequestDTO → Entidade (criando o Endereco junto!)
    public static Cliente toEntity(ClienteRequestDTO dto) {

        Cliente cliente = new Cliente();

        cliente.setNomeCompleto(dto.nomeCompleto());
        cliente.setCpf(dto.cpf());
        cliente.setEmail(dto.email());
        cliente.setTelefone(dto.telefone());
        cliente.setDataNascimento(dto.dataNascimento());

        // Cria o Endereco a partir do DTO aninhado e seta no cliente
        cliente.setEndereco(toEnderecoEntity(dto.endereco()));

        return cliente;
    }

    // Converte Entidade → ResponseDTO
    public static ClienteResponseDTO toResponseDTO(Cliente cliente) {

        return new ClienteResponseDTO(
                cliente.getId(),
                cliente.getNomeCompleto(),
                cliente.getCpf(),
                cliente.getEmail(),
                cliente.getTelefone(),
                cliente.getDataNascimento(),
                toEnderecoDTO(cliente.getEndereco())  // converte o Endereco aninhado
        );
    }

    // ───── ENDERECO (auxiliares) ─────

    // EnderecoDTO → Endereco (entidade)
    public static Endereco toEnderecoEntity(EnderecoDTO dto) {

        Endereco endereco = new Endereco();

        endereco.setCep(dto.cep());
        endereco.setLogradouro(dto.logradouro());
        endereco.setNumero(dto.numero());
        endereco.setComplemento(dto.complemento());
        endereco.setBairro(dto.bairro());
        endereco.setCidade(dto.cidade());
        endereco.setUf(dto.uf());

        return endereco;
    }

    // Endereco (entidade) → EnderecoDTO
    public static EnderecoDTO toEnderecoDTO(Endereco endereco) {

        return new EnderecoDTO(
                endereco.getId(),
                endereco.getCep(),
                endereco.getLogradouro(),
                endereco.getNumero(),
                endereco.getComplemento(),
                endereco.getBairro(),
                endereco.getCidade(),
                endereco.getUf()
        );
    }
}

package br.com.sistemagestao.api.service;

import br.com.sistemagestao.api.dto.cliente.ClienteRequestDTO;
import br.com.sistemagestao.api.dto.cliente.ClienteResponseDTO;
import br.com.sistemagestao.api.mapper.ClienteMapper;
import br.com.sistemagestao.api.model.Cliente;
import br.com.sistemagestao.api.model.Endereco;
import br.com.sistemagestao.api.repository.ClienteRepository;
import br.com.sistemagestao.api.service.ClienteService;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;

import java.util.List;

@Service
@RequiredArgsConstructor
public class ClienteServiceImpl implements ClienteService {

    // Só um Repository! O cascade cuida do Endereco.
    private final ClienteRepository clienteRepository;

    @Override
    public ClienteResponseDTO criarCliente(ClienteRequestDTO dto) {

        // Mapper cria Cliente + Endereco aninhado
        Cliente cliente = ClienteMapper.toEntity(dto);

        // 🌊 Salva o Cliente — o Endereco é salvo junto via cascade!
        clienteRepository.save(cliente);

        return ClienteMapper.toResponseDTO(cliente);
    }

    @Override
    public List<ClienteResponseDTO> listarTodosClientes() {

        return clienteRepository.findAll()
                .stream()
                .map(ClienteMapper::toResponseDTO)
                .toList();
    }

    @Override
    public ClienteResponseDTO buscarClientePorId(Long id) {

        Cliente cliente = clienteRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Cliente não encontrado"));

        return ClienteMapper.toResponseDTO(cliente);
    }

    @Override
    public ClienteResponseDTO atualizarClientePorId(Long id, ClienteRequestDTO dto) {

        Cliente cliente = clienteRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Cliente não encontrado"));

        // Atualiza os campos simples do cliente
        cliente.setNomeCompleto(dto.nomeCompleto());
        cliente.setCpf(dto.cpf());
        cliente.setEmail(dto.email());
        cliente.setTelefone(dto.telefone());
        cliente.setDataNascimento(dto.dataNascimento());

        // Atualiza os campos do Endereco que JÁ EXISTE (mantemos o mesmo id!)
        Endereco endereco = cliente.getEndereco();
        endereco.setCep(dto.endereco().cep());
        endereco.setLogradouro(dto.endereco().logradouro());
        endereco.setNumero(dto.endereco().numero());
        endereco.setComplemento(dto.endereco().complemento());
        endereco.setBairro(dto.endereco().bairro());
        endereco.setCidade(dto.endereco().cidade());
        endereco.setUf(dto.endereco().uf());

        clienteRepository.save(cliente);

        return ClienteMapper.toResponseDTO(cliente);
    }

    @Override
    public void deletarClientePorId(Long id) {

        Cliente cliente = clienteRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                        HttpStatus.NOT_FOUND, "Cliente não encontrado"));

        // 🗑️ Deleta o Cliente — o Endereco é deletado junto via orphanRemoval!
        clienteRepository.deleteById(cliente.getId());
    }
}
🤔
Por que no atualizar mexemos no Endereço existente em vez de criar um novo? Para manter o mesmo endereco_id no banco. Se criássemos um Endereço novo, geraria uma linha nova na tabela endereco e o orphanRemoval apagaria a antiga. Funcionaria, mas seria uma operação a mais. Atualizar a instância existente é mais eficiente! ⚡

import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/clientes")
@RequiredArgsConstructor
public class ClienteController {

    private final ClienteService clienteService;

    @ResponseStatus(HttpStatus.CREATED)

    @ResponseStatus(HttpStatus.OK)
    public ClienteResponseDTO atualizarClientePorId(@PathVariable Long id,
